# AI Prompt Engineering 通用指南

> 整合 Google Gemini、OpenAI GPT、Anthropic Claude 官方指南的通用最佳實踐

## 核心理念：從 Prompt Engineering 到 Context Engineering

從「隨興聊天模式」轉向「工程模式」—— 將提示詞與上下文視為程式碼般精確設計，以獲得更準確的 AI 回覆並降低錯誤率。

**關鍵心態轉變：**
- **Reasoning Model（OpenAI o1、GPT-5 Thinking Mode、Claude Extended Thinking）** = 資深同事：給目標，讓它自己規劃細節。
- **Standard Model（GPT-5 Instant Mode、GPT-4o、Claude Sonnet）** = 初級同事：需要明確指令才能產出特定結果。

### 上下文工程（Context Engineering）

**核心原則：** 找到最小的高信號（high-signal）token 集合，以最大化期望結果的可能性。專注於**提供什麼資訊**，而不僅僅是**如何問問題**。

| 傳統 Prompt Engineering | Context Engineering |
|------------------------|---------------------|
| 專注於「如何問問題」 | 專注於「提供什麼資訊」 |
| 靜態提示詞設計 | 動態資訊策劃 |
| 單次互動優化 | 多步驟工作流優化 |

**實踐要點：**
- **精簡上下文**：每個步驟只提供必要的資訊，避免無關內容分散模型注意力。
- **動態調整**：根據任務的進展，更新、增加或移除上下文內容。
- **最大化信號**：確保提供的每一條資訊都對達成目標有直接幫助。

## 基礎策略 (8 大通用技巧)

所有主流模型（Gemini, GPT, Claude）都適用以下基礎策略。

### 1. 提供清晰、明確的指令

- **使用結構化標籤**：使用 XML 風格標籤 (`<tag></tag>`) 或 Markdown 來分隔提示的不同部分。
- **定義角色**：明確告訴模型它應該扮演什麼角色。
- **明確任務**：使用清晰的動詞來描述任務（「總結」、「翻譯」、「比較」、「程式碼審查」）。
- **定義輸出格式**：明確指定你想要的輸出格式。

**XML 標籤範例：**
```xml
<role>你是一位資深軟體工程師，專精於 TypeScript 和 React。</role>
<context>用戶正在開發一個電商平台的購物車功能。</context>
<task>請審查以下程式碼並提供改進建議。</task>
<constraints>
- 回覆使用繁體中文
- 提供具體的程式碼範例
- 說明每個建議的理由
</constraints>
<code>// 待審查的程式碼</code>
```

---

### 2. 提供參考文本

要求模型根據你提供的參考資料來回答問題，而不是依賴其內部知識。這能大幅減少「幻覺」。

**長文件指令放置策略（Sandwich Method）：**
- 指令放在**開頭**
- 長文件內容放在**中間**
- 指令重複放在**結尾**

**範例：**
```
### 指令
根據下方提供的技術文件，回答使用者的問題。只使用文件內的資訊回答，並附上相關頁碼。

### 技術文件
"""
[...技術文件內容...]
"""

### 問題
根據上述文件，請說明如何設定 API 金鑰？

### 指令（重複）
根據上方提供的技術文件回答，只使用文件內的資訊，並附上相關頁碼。
```

---

### 3. 將複雜任務拆解為簡單子任務

與其讓模型一次處理龐大複雜的任務，不如拆解成一系列更小、更簡單的步驟。

**範例：**
```
請幫我處理這份技術文件，依序執行以下步驟：

**步驟 1：摘要** — 將文件內容精簡為 3-5 個核心重點。
**步驟 2：翻譯** — 將摘要翻譯成流暢的英文。
**步驟 3：格式化** — 將英文摘要轉換為 Markdown 項目清單。

請在每個步驟完成後，等待我的確認再進行下一步。
```

---

### 4. 給模型時間「思考」(Chain of Thought)

要求模型在給出最終答案之前，先進行一步步的分析。能顯著提升推理任務（數學、程式碼、邏輯問題）的表現。

**範例：**
```
在回答問題前，請先遵循以下步驟：
1. **分析問題**：這是關於什麼的？我需要哪些資訊？
2. **建立計畫**：我該如何一步步解決這個問題？
3. **執行計畫**：按照計畫進行計算或推理。
4. **產出答案**：總結我的發現並給出最終答案。

請將思考過程用 `<thinking>` 標籤包裝，最後用 `<answer>` 標籤給出答案。
```

---

### 5. 使用範例 (Few-Shot Prompting)

提供 1-5 個理想的「問題 + 答案」範例，展示你想要的具體格式和內容。通常比冗長的文字描述更有效。

**範例：**
```xml
請將技術術語轉換為一般人能懂的白話文。

<examples>
<example>
  <input>使用 npm install 指令安裝相依套件。</input>
  <output>就像為你的專案安裝外掛一樣，這個指令會自動下載需要的程式元件。</output>
</example>
</examples>

現在請轉換這個句子：
<input>執行 `docker-compose up -d` 來啟動容器化服務。</input>
```

**最佳實踐：**
- 範例應與實際任務高度相關且格式一致。
- 使用正向範例（展示該做什麼）通常優於負向範例。

---

### 6. 正向與負向表述的策略運用

| 表述類型 | 適用情境 | 範例 |
|---------|---------|------|
| **正向表述** | 一般指令、期望行為 | 請將回覆控制在 200 字以內。 |
| **負向表述** | 硬性邊界、內容排除 | 不要包含任何程式碼範例。 |
| **正負結合** | 複雜約束條件 | 請用簡單語言解釋，不要使用技術術語。 |

**正向轉換：** 將模糊的負向指令轉換為具體的正向指令：

| 模糊的負向指令 | 具體的正向指令 |
|---------------|---------------|
| 不要寫太長。 | 請將回覆控制在 200 字以內。 |
| 不要使用專業術語。 | 請使用國小五年級學生能理解的語言。 |

**解釋原因 (Why + What)：** 無論正向或負向表述，解釋「為什麼」都能讓模型更好地遵循指示。

| 只說 What | Why + What（更有效） |
|----------|---------------------|
| 回答要簡短。 | 因為使用者是在手機上閱讀，請將回答控制在 3 段以內。 |

---

### 7. 預填回應開頭

當需要特定的輸出格式時，直接寫下你想要的開頭，模型會自然延續這個格式。

- **JSON 輸出**：以 `{"issues": [` 開頭，模型會自動補完 JSON 陣列。
- **直接進入思考模式**：以 `<thinking>` 開頭，模型會跳過寒暄直接開始分析。

---

### 8. 調整溫度與參數

- **低溫度 (0.2)**：更確定、一致。適用於事實、程式碼生成、精確摘要。
- **高溫度 (0.8-1.0)**：更具創意、多樣性。適用於寫作、腦力激盪。
- **溫度 0**：最確定性輸出，適合需要可重現結果的場景。

| 參數 | 說明 |
|---|---|
| **Max Tokens** | 控制回應的最大長度（100 tokens ≈ 75 個英文單字）。 |
| **Top-P / Top-K** | 另一種控制隨機性的方法。通常建議只修改溫度或 Top-P 其中一個。 |
| **Stop Sequences** | 設定模型在遇到特定文字時自動停止生成。 |

## 進階技巧

### 9. 迭代與改進你的提示

| 策略 | 說明 |
|---|---|
| **重新表述** | 用不同的措詞或方式來表達你的請求。 |
| **改變順序** | 調整提示中指令、上下文和範例的順序。 |
| **增加範例** | 補充更多或更具代表性的 Few-Shot 範例。 |
| **尋求自我修正** | 要求模型「批評」自己的前一個回答，並根據批評進行改進。 |

> **核心心態**：將提示工程視為實驗科學。觀察輸出，形成假設，修改提示，然後再次測試。

---

### 10. 多模態提示

利用現代模型處理圖像、音訊甚至影片的能力。

**圖片分析最佳實踐：**
- **提供工具**：為模型提供 `crop/zoom` 工具，讓它聚焦在圖片的特定區域。
- **明確標示**：處理多張圖片時，用清晰標籤（如 `圖片 A`, `圖片 B`）指代。
- **結合文字**：將圖片與文字指令結合，提出具體問題。

---

### 11. 使用工具與函式

讓模型呼叫外部工具或函式，擴展其能力（即時資訊、外部系統互動、複雜計算）。

**高效工具描述的原則：**

| 原則 | 說明 |
|---|---|
| **專注「做什麼」** | 描述業務目標，而非技術細節。 |
| **明確「何時用」** | 說明觸發工具的具體情境。 |
| **清晰「參數化」** | 清楚解釋每個參數的意義和格式。 |

> 在 system prompt 中鼓勵模型在多個工具之間沒有依賴關係時進行平行呼叫，可以提高效率。

---

### 12. 元提示 (Meta Prompting)

讓 LLM 幫你編寫和優化提示詞。

- **自動化優化**：讓 AI 根據目標生成多個提示詞候選版本。
- **發現盲點**：AI 可能產生人類工程師沒想到的、效果更好的提示詞結構。
- **建立評估**：讓 AI 根據一系列標準來評估不同提示詞的優劣。

**實踐範例：**
```
你是一位頂尖的提示詞工程師。
你的任務是為一個「分析客戶回饋並總結情緒」的 AI 助手，創建一個高效的提示詞。

這個提示詞需要包含：
1. 清晰的角色定義。
2. 結構化的任務描述，要求 AI 識別正面、負面和中性的回饋。
3. 一個 Few-shot 範例，展示理想的輸入和輸出格式 (JSON)。
4. 明確的限制，例如「只能使用提供的回饋內容」。

請為我生成這個提示詞模板。
```
